Canvascope can take action now, and it still can't do your homework

The assistant has been able to answer questions about your courses for a while. The thing it couldn't do was act on the answer. Now it can.

Up to now the actual work was always left to you. Canvascope would read your courses and answer with citations, and then you'd be the one making the to-do, adding the calendar event, or turning a wall of deadlines into some kind of plan. The new agent picks up at that point. You can ask it something like "help me get ready for my biochem class this week," and it'll check your calendar, look up what's due, dig through your own course materials, and hand back a study plan, a set of to-dos, or calendar events.

What changed under the hood is that it works in steps now instead of trying to do everything in one shot. It figures out what it needs, calls a tool to get it, reads the result, then decides on the next move, and repeats that a handful of times until it actually has an answer. You can watch the whole thing in the side panel: Checking your calendar → Looking up deadlines → Drafting a plan, each step flipping from in-progress to done.

I kept the list of things it's allowed to do short on purpose. It can read your active page, your indexed course materials, upcoming deadlines, grades, and your calendar. And it can create study aids: a to-do, a calendar event, a study plan. That's it. There's nothing else on the list.

The line I wasn't willing to cross is submitting your work. It can't. There's no button for it to press, because that capability was never built into the agent in the first place. Beyond that, anything it creates runs through a check that blocks anything resembling a graded submission, every action gets logged, every action can be undone, and one switch pauses the whole thing. Like the rest of Canvascope, it lives inside the extension and only reaches out when a task genuinely needs it.

It'll also check in once a day, if you want it to, with a short rundown of what's coming up. So instead of opening to an empty search box, you open to a bit of a head start. That's the direction I'm taking this: an assistant that helps you actually get moving, without ever crossing the one line that matters. It helps you learn. It doesn't do the graded work for you.

Lectra runs a real terminal, git, and Python on the iPad now

Notebooks got code into Lectra. This is about giving that code somewhere to actually live.

A lot of student work these days is just a repo. A data science project, a problem set that's really a few scripts, a lab notebook that you'd like to have version history for. So Lectra got a full Projects workspace, the idea being that this work can sit on the iPad instead of bouncing between a laptop and some cloud IDE. You can clone a GitHub repo, edit files right next to your PDFs and notebooks, run Python, and commit and push, all without leaving the app.

I want to be clear this isn't a dressed-up text box. iOS gives you no fork/exec, so the terminal is a POSIX-style shell I wrote from scratch: a real lexer, parser, pipelines, redirection, globbing, and around two dozen built-in commands, all operating on the app's own files. Git is the genuine article too. isomorphic-git runs in a hidden web layer with its filesystem and network bridged down to native Swift, which means clone, commit, pull, and push are really talking to GitHub over authenticated HTTPS. Python runs through an embedded runtime, so python script.py does the thing you'd expect it to.

The detail I'm proudest of is that all of it shares one sandboxed filesystem. The terminal, the editor, and git are looking at the exact same files, so git status shows precisely what the editor just saved. No hidden copies, no sync step in between. Every path stays inside the app's sandbox, and your GitHub token only ever gets attached to requests going to GitHub.

There's also a one-tap Sync. It stages your latest edits, commits, pulls, and pushes in a single pass, so the thing you do constantly, save my work back to GitHub, is one button rather than five commands. Same thing I was going for with notebooks: if code shows up in your coursework, it should feel like part of the workspace, not the reason you have to go find your laptop.

Python notebooks are first-class Lectra documents now

Lectra started with PDFs. The problem is that almost no class actually fits in one file format.

Think about a data science assignment: there's the problem set PDF, a couple of markdown notes, and a short Python analysis. Or a biology lab, with a protocol, an annotated paper, and some quick calculation off to the side. In practice all of those pieces end up scattered across a PDF editor, a notebook site, a notes app, and wherever your file manager dumped the download.

Native .ipynb support is my attempt to pull that back into one place. You can create a Python notebook from the same +New menu you already use for notebooks, whiteboards, folders, and PDF imports, and it lands in the current folder right next to the rest of the assignment, not off in some separate developer corner of the app.

And it's a working notebook, not a preview. Lectra edits code and markdown cells, runs Python, captures the output, and holds state between cells. I'm not trying to turn the iPad into a desktop IDE. I just want the small bits of code that show up in actual coursework to feel like they belong with the reading and the annotation and the notes.

The editor details are where a notebook either feels real or feels fake, so those got attention too: Python syntax highlighting, auto-indent after a colon, keyboard shortcuts for running cells and moving between them. Little things, but they're what keep it from feeling like a text box pretending to be a notebook.

It's not finished. Notebook thumbnails still need their own look, deleting a notebook should clean up the files underneath it, and there's more to add. But the base is there now, and that's what I was after: PDFs, notes, whiteboards, folders, imports, and Python notebooks all sitting in one Lectra library.

June 14, 2026 · Engineering

Safer production routing for DropBridge

Before I touched any delivery behavior, I had to fix the DropBridge backend's history. Ten production migrations had drifted, so I reconstructed them from the remote migration table until local development lined up with the real Supabase project again.

Once the history was clean, I added sender_device_id to uploads. It gives DropBridge a reliable way to tell the device that sent a file apart from the device that's supposed to receive it. That distinction is what makes instant delivery confirmation possible, and it's how you stop a file from getting picked up twice or landing on the wrong machine.

None of this is visible. It's backend plumbing. But it protects the part students do feel: a PDF leaves Lectra, wakes Canvascope, downloads on the right computer, and reports back, with no guessing about whether it worked.

June 13, 2026 · Security

Real account-event protection in Canvascope

Canvascope now has Cross-Account Protection wired up through Google's RISC security event stream. When Google reports a high-risk event on an account, Canvascope validates the signed event, dedupes it, maps it to the right Supabase user, and revokes their active sessions.

If the event is an account-disabled one, it also sets a sign-in block that the Supabase custom access token hook enforces on both sign-in and token refresh. Re-enable the account later and the block clears. I made the hook fail-open on purpose: if something inside it errors, I'd much rather it let people through than have one internal bug lock out the entire user base.

It's not a feature anyone will see in the UI. But it's the kind of groundwork a student workspace needs in place before it starts handling more files, more identity, and more device-to-device traffic.
